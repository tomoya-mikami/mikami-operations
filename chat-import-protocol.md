# 他チャット情報取り込みプロトコル

最終更新: 2026-05-25

## 結論

他チャットの内容は、本文を丸ごと貼らず、各チャット側で「引き継ぎ要約」に圧縮してから `mikami-operations` に取り込む。

理由:

- 長い本文をそのまま貼ると、受け側チャットの文脈上限を圧迫する。
- 重要な決定、未完了タスク、成果物、次アクションだけ残せば再開できる。
- 財務、税務、プラットフォーム規約、価格などは、要約後に必要部分だけ原資料や最新情報で確認すればよい。

## 基本方針

- 古いチャット本文はGoogle Drive等に保管してもよいが、GitHubへ丸ごと保存しない。
- GitHubには「業務に使う要約」「決定事項」「未完了タスク」「ファイル所在」だけを保存する。
- パスワード、口座番号、マイナンバー、顧客個人情報、高機密資料は取り込まない。
- 数値、法律、税務、金融、eBay等の規約、料金は「要検証」として扱い、正式判断前に最新ソースで確認する。
- 受付チャットは一時置き場。正本は `chat-import-ledger.md` と各Markdown管理ファイルに置く。
- Fayは取り込まれた要約を見て、重要度・進捗・反映先・Alfへ渡すべきタイミングを判断する。

## 受付チャット運用

### 推奨

まずは受付チャットを1つにする。

推奨チャット名:

`旧チャット取り込み - 受付`

理由:

- 三神さんが貼る場所を迷わない。
- Fayが全体の重複、重要度、優先順位を判断しやすい。
- 後から `chat-import-ledger.md` で一覧化すれば、チャットが1つでも情報は整理できる。

### 分けた方がよい場合

以下のように量が多い、または分野が明確に違う場合は受付チャットを分けてもよい。

- `旧チャット取り込み - 財務`
- `旧チャット取り込み - リサーチ`
- `旧チャット取り込み - 送料・ツール`
- `旧チャット取り込み - 国内販売`

ただし、複数チャットに分ける場合でも、集計の正本はチャット本文ではなく `chat-import-ledger.md` と管理ファイルにする。

### 集計担当

- 全体集計: Fayの主チャット
- ファイル整理・大量反映: Alf
- 正本: `mikami-operations` 内のMarkdownファイル

チャット同士は互いの本文を自動で読めないため、複数受付チャットに分けた場合も、各チャットの結果を必ず `chat-import-ledger.md` へ戻す。

## Fayが判断すること

Fayは、取り込まれた要約ごとに以下を判断する。

| 判断項目 | 見るポイント |
|---|---|
| 重要度 | 売上、資金繰り、法務/税務リスク、業務停止リスク、今後の再利用性 |
| 緊急度 | 今週中に必要、融資/出品/支払い/納期に影響する、期限がある |
| 進捗 | 未着手、進行中、確認待ち、完了、保留、古い可能性あり |
| 反映先 | `handoff.md`, `projects.md`, `delegated-tasks.md`, `意思決定ログ.md`, テーマ別メモ |
| Alf依頼要否 | 3-5件以上たまった、ファイル反映が多い、重複整理が必要、commit/pushが必要 |
| 要確認 | 機密情報、古い数値、税務/金融/法務/規約/料金など最新確認が必要な情報 |

## Alfへ渡す目安

Fayは以下のどれかに当てはまったら、Alfへ作業パケットを作る。

- 取り込み要約が3-5件たまった。
- 1件でもファイル反映量が多い。
- 複数チャット間で内容が重複している。
- `projects.md` や `delegated-tasks.md` など複数ファイル更新が必要。
- 反映後にcommit/pushして正本化した方がよい。
- 受付チャットが長くなってきた。

Fayはチャットがいっぱいになる前に、以下を三神さんへ知らせる。

```text
取り込みが増えてきました。ここで一度Alfへ渡して、台帳化・ファイル反映・commit/pushまで進めます。
```

## 推奨手順

1. 旧チャット側で、まず「目次化プロンプト」を実行する。
2. 目次を見て、重要テーマごとに「詳細引き継ぎプロンプト」を実行する。
3. 出力をFayへ貼る。量が多い場合は、Alfへ渡してファイル化・整理する。
4. Fayが内容を分類し、必要に応じて以下へ反映する。
   - `handoff.md`: 現在地、次アクション
   - `projects.md`: プロジェクト状態
   - `delegated-tasks.md`: 分割作業、担当、進捗
   - `意思決定ログ.md`: 決定事項と理由
   - 個別メモ: 財務、送料、リサーチ、国内販売などのテーマ別ファイル

## `context_length_exceeded` が出た時の対処

旧チャット側やremote compactで以下のエラーが出た場合は、チャット全体の圧縮をあきらめ、テーマ単位で分割して取り込む。

```text
Your input exceeds the context window of this model.
```

対処:

1. 旧チャット全体を一度に要約しない。
2. 目次案がある場合は、目次の1テーマだけを対象にする。
3. 1回の出力は3,000-5,000字程度を目安にする。
4. 長い場合は `Part 1/3`、`Part 2/3` のように分ける。
5. 決定事項、未完了タスク、成果物、要検証事項だけを優先し、会話の経緯は省く。
6. 機密情報や個人情報は出力しない。

旧チャット側へ貼る短縮プロンプト:

```text
このチャット全体の圧縮で context_length_exceeded が出ました。
チャット全体ではなく、以下のテーマだけを短く引き継ぎ要約してください。

対象テーマ:
（ここにテーマ名）

出力は最大5,000字以内。
長くなる場合は Part 1/3 のように分割してください。

含めるもの:
- 現在地
- 決定事項
- 未完了タスク
- 成果物・参照先
- 要検証事項
- Fayへ反映すべき内容

含めないもの:
- 会話全文
- 雑談や経緯の細部
- パスワード、口座番号、マイナンバー、顧客個人情報
- APIキー、ログイン情報、個人メールアドレス
```

### 分割プロンプトでも再失敗する場合

同じ旧チャットで再度 `context_length_exceeded` が出る場合、そのチャットはAI側で本文を読み直して要約する段階を超えている可能性が高い。

その場合の方針:

1. 旧チャットへ追加指示を送るのを止める。
2. すでに取り出せた目次案を正本候補として使う。
3. 詳細はチャット本文ではなく、作成済みファイル、Google Sheets、GAS、CSV、docx、xlsx、PDF、手元のメモから復元する。
4. 復元作業はFayではなくAlfへ渡す。
5. 復元時も、APIキー、ログイン情報、顧客個人情報、契約条件詳細は保存しない。

この状態になったら、Fayは `alf-old-chat-artifact-recovery-packet.md` を作成し、Alfに成果物ベース復元を依頼する。

## 受付チャット初回プロンプト

このプロンプトを受付チャットの最初に貼る。以後は、旧チャット側で作った要約を貼っていくだけでよい。

```text
Fayとして、旧チャット情報の取り込み受付をしてください。

このチャットの役割:
- 旧チャットから出力された要約を受け取る
- 内容の重要度、緊急度、進捗、反映先を判断する
- これまで行った作業をリスト化・見える化する
- 3-5件たまった、または反映量が多いと判断したら、Alfへ渡す作業パケットを作る
- チャットが長くなってきたら、いっぱいになる前に三神さんへ知らせる

大前提:
- 三神さんは、これ以降、旧チャットの引き継ぎ要約をそのまま貼っていく
- 毎回「旧チャットからの引き継ぎ要約です」と細かく指示しなくても、Fayが取り込み要約として扱う
- ただし、可能なら貼り付け冒頭に「財務」「リサーチ」「送料」「国内販売」などの分類名だけ付ける

判断してほしいこと:
- 重要度: 高/中/低
- 緊急度: 高/中/低
- 状態: 未着手/進行中/確認待ち/完了/保留/古い可能性あり
- 反映先: `handoff.md`, `projects.md`, `delegated-tasks.md`, `意思決定ログ.md`, テーマ別メモ
- Alfへ渡すべきか
- 機密情報や要最新確認の有無

保存・反映方針:
- 正本はチャット本文ではなく、`mikami-operations` のMarkdownファイル
- 取り込み一覧は `chat-import-ledger.md` に集約する
- パスワード、口座番号、マイナンバー、顧客個人情報は保存しない
- 税務、金融、法律、eBay等の規約、送料、価格は必要に応じて「要最新確認」とする

出力は毎回、まず短く以下の形式にしてください。

取り込み判定:
- 分類:
- 重要度:
- 緊急度:
- 状態:
- 反映先:
- Alf依頼:
- 要確認:
- 次アクション:
```

## 旧チャット用: 目次化プロンプト

```text
このチャットの内容を、新しい秘書ハブ `mikami-operations` へ引き継ぐ準備をします。

会話全文を再掲せず、まず「目次」だけ作ってください。

目的:
- 重要テーマを漏らさず把握する
- どの部分を詳細要約すべきか決める
- 受け側チャットの文脈上限を圧迫しない

出力形式:

# チャット目次

## 1. このチャットの役割
- 何を担当していたチャットか

## 2. 主要テーマ一覧
| No | テーマ | 重要度 | 状態 | 200字以内の概要 |
|---|---|---|---|---|

## 3. 詳細引き継ぎすべきテーマ
- 優先順に並べる

## 4. 取り込み注意
- 機密情報、個人情報、古い可能性がある情報、検証が必要な情報

ルール:
- パスワード、口座番号、マイナンバー、顧客個人情報は出力しない
- 日付が分かるものは日付を書く
- 決定済み、作業中、保留、古くなった情報を区別する
- 不明な点は推測せず「不明」と書く
```

## 旧チャット用: 詳細引き継ぎプロンプト

```text
このチャットの内容を、新しい秘書ハブ `mikami-operations` へ引き継ぐため、以下のテーマだけを詳細要約してください。

対象テーマ:
（ここに目次のテーマ名を書く）

出力形式:

# 引き継ぎ要約: （テーマ名）

## 1. このテーマの目的

## 2. 現在地
- どこまで進んだか
- 何が未完了か

## 3. 決定事項
| 日付 | 決定 | 理由 | 根拠/関連ファイル |
|---|---|---|---|

## 4. 未完了タスク
| 優先度 | タスク | 担当候補 | 次アクション | ブロッカー |
|---|---|---|---|---|

## 5. 成果物・参照先
| 種類 | 名称/URL/ファイル名 | 用途 | 状態 |
|---|---|---|---|

## 6. 重要な前提・数値
| 項目 | 内容 | 確定/要検証 | 備考 |
|---|---|---|---|

## 7. Fayへ反映すべき内容
- `handoff.md` に入れる内容
- `projects.md` に入れる内容
- `delegated-tasks.md` に入れる内容
- `意思決定ログ.md` に入れる内容
- 個別ファイルを作るべき内容

## 8. 注意事項
- 機密情報
- 古い可能性がある情報
- 正式判断前に確認が必要な情報

ルール:
- 会話全文の再掲は禁止
- パスワード、口座番号、マイナンバー、顧客個人情報は出力しない
- 財務、税務、法律、プラットフォーム規約、価格、送料は、必要に応じて「要最新確認」と明記する
- 推測で補完せず、不明点は「不明」と書く
- 出力が長くなる場合は「Part 1/3」のように分割する
```

## 財務担当チャット用プロンプト

```text
この財務担当チャットの内容を、新しい秘書ハブ `mikami-operations` へ引き継ぐために要約してください。

特に以下を整理してください。

- 融資相談の現在地
- 資金繰り上の課題
- 税理士・会計事務所とのやり取り
- 必要資料
- 決算、月次、試算表、売上、利益、在庫、借入などの論点
- 未完了タスク
- 次に三神さんが確認すべきこと

出力形式は「詳細引き継ぎプロンプト」と同じにしてください。

注意:
- 口座番号、ログイン情報、個人番号、顧客個人情報は出力しない
- 金額や条件は、正式資料で再確認が必要な場合は「要原資料確認」と書く
- 税務・融資の判断は、この要約だけで確定せず、必要に応じて税理士・金融機関・最新資料で確認する前提にする
```

## Fay側に貼るときのプロンプト

```text
Fay、以下は旧チャットからの引き継ぎ要約です。

やってほしいこと:
1. 機密情報や保存不可情報が含まれていないか確認
2. 内容を分類
3. `handoff.md`、`projects.md`、`delegated-tasks.md`、`意思決定ログ.md`、または個別メモに反映すべき項目を提案
4. 量が多い場合は、Alfへ渡す作業パケットを作成

まずは保存前に、現在地・反映先・未確認事項だけ短くまとめてください。

--- 引き継ぎ要約ここから ---
（ここに旧チャットの要約を貼る）
--- 引き継ぎ要約ここまで ---
```

※受付チャットに初回プロンプトを入れた後は、この毎回用プロンプトは省略してよい。

## Alfへ渡す場合の依頼文

```text
この作業はアルフで進めてください。
作業場所:
/Users/mikami/Documents/GitHub/mikami-operations-official

作業内容:
旧チャットから出力した引き継ぎ要約を、`mikami-operations` の管理ファイルへ整理・反映してください。

入力:
（ここに引き継ぎ要約、または保存ファイル名を書く）

反映先候補:
- `handoff.md`
- `projects.md`
- `delegated-tasks.md`
- `意思決定ログ.md`
- 必要に応じたテーマ別メモ

完了条件:
- 機密情報や保存不可情報が入っていないことを確認した
- 必要な管理ファイルを更新した
- 不明点、要確認事項、次アクションを残した
- `git status` で変更内容を確認した
- 問題なければ commit した
- `git push origin main` でGitHubへ反映した
- 最後に、何を反映したかFayへ短く報告した

注意:
- パスワード、口座番号、マイナンバー、顧客個人情報は保存しないでください
- 削除や不可逆な整理はしないでください
- 判断が必要な場合は作業を止めて三神さんに短く質問してください
```
