# カーパーツ価格監視・販売価格見直しロードマップ

最終更新: 2026-05-26

## 結論

カーパーツの最優先改善は、モノタロウ価格変動と販売終了を検知し、赤字化する前に販売価格見直し候補へ上げること。

最初から完全自動の価格改定までは行わず、まずは既存入力シートを壊さない別タブ方式で、価格監視、在庫/取扱状態確認、利益率アラート、見直し候補抽出までを1週間以内に動く形へ持っていく。

## 前提

- 現在の対象件数は約5,000件。
- まずは2週間で1周を標準にする。
- 1周ペースは設定タブで変更できるようにする。
- モノタロウは高頻度巡回せず、低頻度・分散実行・手動確認併用にする。
- ログイン価格、キャンペーン価格、ページ構造変更、取扱終了表示には注意する。
- 既存シートの削除、既存列の破壊、価格改定の即時反映、出品取り下げの即時実行はしない。
- Google Sheets ID、認証情報、サービスアカウント情報、Secrets値は正本Markdownへ保存しない。

## 現行リサーチ入力フロー

三神さん説明ベースの現行入力:

| 列 | 内容 | 役割 |
|---|---|---|
| B | モノタロウ商品ページURL | 仕入価格確認元 |
| C | モノタロウ税別価格 | 価格計算の元値 |
| D | 入数 | 1袋2個などの内容数量 |
| E | C列価格 x 85% | 部品商経由の想定仕入価格 |
| F/G | ヤフオク関連価格 | 国内側の参考価格 |
| H | eBay基準店舗の商品URL | 価格設定の基準 |

価格設定ロジック:

- 部品商と関係がある基準セラーが同型番を出している場合、三神さん側の「商品価格 + 送料」の総額が、基準セラーの総額を下回らないようにする。
- 基準セラーが出していない商品は、他セラー価格と利益率を見ながら任意に値付けする。
- 商品価格と送料の内訳は完全一致でなくてよいが、総額の基準は守る。

## 解決したい課題

| 課題 | 影響 | 先にやる度 |
|---|---|---|
| モノタロウ価格の値上がりに気づけない | 売れてから赤字に気づく | 最優先 |
| 在庫切れ・取扱終了に気づけない | 売れた後に仕入不可 | 最優先 |
| 利益率低下が見えない | 価格改定漏れ | 最優先 |
| サイズ/重量不明のままShipping Policyを選ぶ | 送料赤字 | 高 |
| 基準セラー総額を下回る | 価格秩序が崩れる | 高 |
| 既存出品全体の価格/Policy見直しが未完了 | 現在出品中の商品に赤字リスク | 高 |

## 推奨シート構成

既存の入力用シートは触りすぎず、追加タブで管理する。

| タブ | 目的 | 備考 |
|---|---|---|
| `設定` | 巡回周期、1回処理件数、利益率閾値、価格上昇閾値を管理 | 三神さんが後から調整する場所 |
| `価格監視マスター` | URL、SKU、品番、前回価格、今回価格、状態、次回確認日を管理 | 既存入力シートから参照または同期 |
| `価格履歴` | 取得結果を追記保存 | 上書きせず、後で価格変動を追えるようにする |
| `販売価格見直し候補` | 値上がり、利益率低下、取扱終了、在庫切れを一覧化 | 人間が判断する作業リスト |
| `手動確認キュー` | 自動取得失敗、価格不明、複数候補、ログイン要確認を集約 | 外注/社内確認に渡せる |
| `Shipping見直し候補` | 重量/サイズ不明、Policy不適合候補を集約 | Phase2で作る |
| `送料差額チェック` | 推定送料と実送料の差額をSKU/注文単位で確認 | 追加MVPで作る |
| `関税チェック` | 想定関税額と請求関税額の差額を確認 | 追加MVPで作る |
| `Chatwork情報取込` | 品切れ、値上げ、廃番、納期未定の投稿を抽出 | まず手動貼付 |
| `出品反映候補` | 価格、送料、関税、Chatwork情報を1つの候補リストに集約 | 自動反映はしない |

## 設定タブ案

| 設定名 | 初期値 | 意味 |
|---|---:|---|
| `PRICE_CHECK_CYCLE_DAYS` | 14 | 何日で1周するか |
| `ACTIVE_ITEM_COUNT` | 自動計算 | 監視対象件数 |
| `TARGET_ROWS_PER_DAY` | 自動計算 | `ACTIVE_ITEM_COUNT / PRICE_CHECK_CYCLE_DAYS` |
| `MAX_ROWS_PER_RUN` | 80 | 1回のGAS実行で見る最大件数 |
| `RUNS_PER_DAY` | 自動計算 | 1日に必要な実行回数 |
| `MIN_SLEEP_MS` | 2000 | アクセス間隔の最低待機 |
| `PRICE_UP_ALERT_RATE` | 3% | 値上げ率アラート |
| `PRICE_UP_ALERT_YEN` | 100 | 値上げ額アラート |
| `MIN_MARGIN_RATE` | 20% | 利益率下限 |
| `PURCHASE_FACTOR` | 0.85 | 部品商想定仕入係数 |
| `AUTO_UPDATE_SELL_PRICE` | false | 初期は自動価格改定しない |
| `AUTO_END_LISTING` | false | 初期は自動取り下げしない |
| `SHIPPING_DIFF_ALERT_YEN` | 1000 | 送料差額アラート |
| `SHIPPING_DIFF_ALERT_RATE` | 20% | 送料差額率アラート |
| `DUTY_DIFF_ALERT_YEN` | 1000 | 関税差額アラート |
| `DUTY_DIFF_ALERT_RATE` | 20% | 関税差額率アラート |

5,000件を14日で回す場合:

- 1日あたり約358件。
- 1回80件なら、1日5回実行で達成できる。
- 30日で1周に落とす場合は、1日約167件。
- 周期は `PRICE_CHECK_CYCLE_DAYS` を変えるだけで調整できるようにする。

## アラート条件

| 条件 | 表示 |
|---|---|
| 今回価格 > 前回価格 | 価格セルを赤、見直し候補 |
| 今回価格 < 前回価格 | 価格セルを青または緑 |
| 価格上昇率が閾値以上 | 見直し候補 |
| 利益率が下限未満 | 見直し候補 |
| 在庫切れ / 取扱終了 / 納期未定 | 出品停止または要確認 |
| 自社総額が基準セラー総額を下回る | 価格調整候補 |
| 取得失敗が連続 | 手動確認キュー |
| 実送料が推定送料を大きく超過 | Shipping Policy変更候補 |
| 請求関税が想定関税を大きく超過 | 関税/利益率確認候補 |
| Chatworkで品切れ/値上げ/廃番情報あり | 出品反映候補 |

## 技術方針

### Phase 1: モノタロウ価格・状態監視

- B列のモノタロウURLを起点に確認する。
- 型番検索の自動化は後回しにし、まず既存URLの価格変動検知に集中する。
- 取得できるのは公開ページ上の表示価格を前提にする。
- ログイン価格、キャンペーン適用後価格、部品商の実価格は自動取得対象にしない。
- 価格取得に失敗した行は上書きせず `要手動確認` にする。

### Phase 2: 利益率・販売価格見直し

- 今回価格 x 85% を想定仕入価格として再計算する。
- eBay販売価格、送料、手数料、送料Policyを組み合わせて利益率を計算する。
- 最初は価格改定CSVを自動出力するだけにし、実際の改定は三神さん確認後にする。

### Phase 3: Shipping Policy見直し

- 商品名、型番、モノタロウ情報、画像/説明から重量・サイズを推定する。
- AI推定は「確定」ではなく、信頼度付きの候補として扱う。
- Shipping Policy候補を表示し、重い/大きい/不明の商品は要確認にする。

### Phase 4: 既存出品の一括見直し

- 現在出品中SKUと価格監視マスターを突合する。
- 赤字化、取扱終了、送料Policy不適合、基準セラー総額下回りを抽出する。
- Sellsta/eBay更新CSVは候補として出力し、自動反映しない。

### Phase 5: 送料・関税・Chatwork情報の反映

- `送料差額チェック` で、推定送料と実送料を比較する。
- `関税チェック` で、想定関税額と請求関税額の差額を比較する。
- `Chatwork情報取込` で、品切れ、値上げ、廃番、納期未定の投稿から品番候補を抽出する。
- すべて `出品反映候補` に集約し、価格改定/出品停止/Policy変更/仕入先確認の候補にする。
- Chatwork APIトークンや認証情報はGitHubやシート本文に保存しない。

## 1週間ロードマップ

| 日 | ゴール | 作業 |
|---|---|---|
| 1日目 | 現行シート構造の読み取り確認 | 入力用タブ、列、既存関数、対象件数、SKU/品番/URL列を確認。直接上書きしない |
| 2日目 | 監視用タブ作成 | `設定`、`価格監視マスター`、`価格履歴`、`販売価格見直し候補` の最小構成を作る |
| 3日目 | 価格取得MVP | 10件、50件、100件の順にテスト。価格、状態、取得失敗を記録 |
| 4日目 | 2週間巡回設定 | 5,000件/14日の分散実行設定。途中停止・続き再開・ログ保存を実装 |
| 5日目 | 利益率アラート | 想定仕入価格、利益率、見直し候補、出品停止候補を自動表示 |
| 6日目 | 既存出品との突合準備 | 出品中SKU、販売価格、送料、Shipping Policyを取り込み、監視マスターと接続 |
| 7日目 | 見直しレポート作成 | 価格改定候補、出品停止候補、Shipping Policy確認候補を一覧化 |

追加課題を含む最短版:

| 2日目 | 送料差額MVP | 推定送料、実送料、差額、差額率、Policy見直し候補を出す |

## 最短で効果を出す順番

1. モノタロウURLから価格と状態を確認する。
2. 値上がり、取扱終了、在庫切れ、取得失敗を可視化する。
3. 仕入価格変更後の利益率を再計算する。
4. 販売価格見直し候補を出す。
5. 現在出品中の商品へ突合する。
6. Shipping Policyの重量/サイズ見直しを進める。
7. 送料差額、関税差額、Chatwork情報を `出品反映候補` に集約する。
8. 価格改定CSVや出品停止候補CSVを作る。

## Alf向け作業メモ

実装はAlfで進めるのが適切。

理由:

- 5,000件規模のシート読み取り、GAS作成、テスト、ログ確認が必要。
- 夜間の分散実行やトリガー確認が必要。
- 既存ツール `carparts-manager`、Sellsta CSV、Shipping Policyマスターとの接続が必要。

Alf側の禁止事項:

- 既存入力シートの列削除、既存関数の上書き。
- モノタロウへの高頻度アクセス。
- 自動価格改定や自動出品取り下げ。
- Google Sheets ID、認証JSON、Secrets値のGitHub保存。
- Google Driveファイルの削除、移動、リネーム。
