# 引き継ぎリスク台帳

最終更新: 2026-05-24

## 目的

スマホ、ノートPC(Fay)、デスクトップPC(Alf)、Codex、Claudeをまたぐ運用で、チャットや作業内容が迷子になるリスクを先回りして管理する。

## 基本方針

- GitHubリポジトリ `mikami-operations` を共通作業台の正本にする。
- 新チャット開始時は `START_HERE.md` を入口にする。
- 毎回GitHub同期を必須にしない。ただし端末をまたぐ作業では同期を必須にする。
- Fayは司令塔、Alfは重い実作業・commit/push担当。
- 重要な決定、次アクション、分割作業は会話だけに残さない。
- 安全な保存・台帳更新・commit/pushは、毎回確認せず進める。
- 削除、不可逆操作、個人情報/機密情報へのアクセスや保存、Google Drive内の削除・移動は必ず確認する。

## リスクと対策

| ID | リスク | 起きる場面 | 影響 | 対策 |
|---|---|---|---|---|
| R-001 | 新チャットに前の会話本文が自動継承されない | チャット上限、別チャット移行 | 役割・決定・未完了タスクが抜ける | `START_HERE.md` を入口にし、`handoff.md`、`意思決定ログ.md`、`delegated-tasks.md` を常時更新する |
| R-002 | 最新状態がノートPCとデスクトップPCでズレる | Alfがpush後、Fayが古い状態を見る | 古い指示で作業する | 端末をまたぐ前後は必ずFetch/Pull。Alfは作業後push、Fayは確認前pull |
| R-003 | FayとAlfが同じファイルを同時編集して衝突する | 並行作業 | マージ衝突、上書き | Fayが台帳で作業所有者と対象ファイルを決める。同じファイルを同時に触らない |
| R-004 | GitHub CLI認証切れでpushできない | 新端末、トークン失効 | 保存はあるが共有されない | `gh auth status` で確認。切れたら `gh auth login --hostname github.com --git-protocol https --web` で再認証。GitHub Desktop pushを予備手段にする |
| R-005 | スマホからローカルファイルやCodexを直接操作できない | 外出中 | 指示は出せるが実行できない | スマホではGitHub上の `START_HERE.md` / `handoff.md` を参照し、Alfへ渡す依頼文を送る。必要ならChrome Remote DesktopでAlfを操作する |
| R-006 | デスクトップPCがスリープ/オフラインでAlfが動かない | 外出中、長時間作業 | 作業が止まる | デスクトップPCを電源接続、スリープ抑制、ネット接続維持。遠隔操作手段を事前に確認する |
| R-007 | 機密情報をGitHubへ保存してしまう | プロフィール、顧客情報、会計資料 | 情報漏えい | 口座番号、パスワード、マイナンバー、顧客個人情報はGitHub禁止。資料本体はGoogle Drive、GitHubは台帳と引き継ぎだけ |
| R-008 | Google Drive資料とGitHub台帳の役割が混ざる | 資料整理、ファイル作成 | 正本が分からなくなる | Google Driveは資料保管、GitHubは作業ログ・引き継ぎ台帳と明確化する |
| R-009 | チャットが長くなりすぎて突然止まる | 長時間相談、大量貼り付け | 最後の判断が消える | 重要決定が出たら即 `意思決定ログ.md`、次アクション変更時は `handoff.md`。長くなったら早めに移行を提案する |
| R-010 | 作業結果がFayへ戻らない | Alfや別チャットで作業 | 司令塔が最新状況を把握できない | 作業完了条件に「handoff更新、commit、push、短い報告」を必ず入れる |
| R-011 | ファイル名や日本語名が環境により化ける | Git CLI、ログ表示 | 見落とし、操作ミス | 重要な運用ファイルは必要に応じて英数字名を併用する。日本語ファイルを扱う時は `rg --files` で実名確認 |
| R-012 | 調査情報が古くなる | 料金、規約、税務、プラットフォーム仕様 | 誤判断 | 公式・最新情報が必要なものは都度確認。古い決定は `意思決定ログ.md` に根拠日付を残す |
| R-013 | どのAI/端末が担当か曖昧になる | 複数AI、複数チャット | 重複作業、漏れ | `delegated-tasks.md` に担当、場所、目的、成果物、次アクション、反映状況を記録する |
| R-014 | ローカルの秘書プロジェクトとGitHub共通作業台が分岐する | Fay側でファイル更新後 | どちらが最新か不明 | `mikami-operations` を正本にする。重要更新後はGitHub側へ反映し、commit/pushする |
| R-015 | 確認が多すぎて作業が進まない | 保存、台帳更新、commit/pushのたびに確認する | 運用が重くなり、記録漏れが増える | 安全な保存・更新・commit/pushはFayが自動実行。削除・不可逆操作・機密情報関連だけ確認する |
| R-016 | FayまたはAlf片方のチャットがいっぱいになる | 長時間作業、重い調査、大量貼り付け | 片方の作業文脈が止まる | もう片方がGitHub正本を更新し、`alf-restart-prompt.md` / `fay-restart-prompt.md` で新チャット用プロンプトを作る |

## 標準チェックリスト

### Fayで新チャット開始

- `START_HERE.md` を読む。
- `handoff.md` で現在地を確認する。
- 必要なら `delegated-tasks.md` と `projects.md` を確認する。
- 端末をまたぐ直前/直後だけGitHub同期を確認する。

### Alfへ作業依頼

- 作業場所を明記する。
- 入力ファイル、期待成果物、禁止事項、完了条件を明記する。
- 作業後は `handoff.md` 更新、`git status`、commit、push、短い報告を必須にする。

### 片方のチャットがいっぱいになった時

- Alfがいっぱいになったら、Fayが `alf-restart-prompt.md` を使ってAlf新チャット用プロンプトを作る。
- Fayがいっぱいになったら、Alfが `fay-restart-prompt.md` を使ってFay新チャット用プロンプトを作る。
- どちらもGitHub正本の `handoff.md`、`projects.md`、`delegated-tasks.md` を更新してから移行する。

### スマホから指示

- 長文説明ではなく、`START_HERE.md` と該当ファイル名を指定する。
- 重い作業はAlfへ回す。
- Fayには司令塔として、依頼文作成・優先順位整理・進捗確認を頼む。
