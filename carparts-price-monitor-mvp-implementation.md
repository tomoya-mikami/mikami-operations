# カーパーツ価格監視MVP 実装メモ

最終更新: 2026-05-27

## 結論

本番シートへ直接貼れるGAS MVPを作成した。

GASファイル:

- `gas/carparts_price_monitor_mvp.gs`
- `gas/carparts_cost_control_mvp.gs`

このMVPは、既存入力シートを削除・上書きせず、新規タブを作り、モノタロウ価格確認結果を候補表示するところまでを担当する。

追加の `carparts_cost_control_mvp.gs` は、送料差額、関税差額、Chatwork貼付情報、AI入力補助、出品反映候補を扱う。こちらも候補表示のみで、自動価格改定や自動出品停止は行わない。

## 作成するタブ

| タブ | 役割 |
|---|---|
| `設定` | 巡回周期、1回処理件数、待機時間、価格上昇閾値、列番号を調整 |
| `価格監視マスター` | 監視対象URL、前回価格、今回価格、状態、次回確認日を管理 |
| `価格履歴` | 価格確認結果を追記保存 |
| `販売価格見直し候補` | 値上がり、利益率低下、取扱終了などの対応候補 |
| `手動確認キュー` | URL不明、価格不明、取得失敗など手動確認が必要な行 |
| `コスト監視設定` | 送料差額、関税差額、Chatworkキーワードの閾値 |
| `eBay注文取込` | eBay注文データを貼り、注文国/売上/送料を整理 |
| `部品商請求明細` | 部品商の直送注文ごとの仕入価格/送料を確認 |
| `送料差額チェック` | 推定送料と実送料の差額確認 |
| `関税チェック` | 届いた関税請求書の請求額と想定関税額の差額確認 |
| `注文コスト突合` | eBay注文、部品商請求、関税請求を突合して粗利/利益率を確認 |
| `Chatwork情報取込` | 品切れ、値上げ、廃番、納期未定情報の貼付解析 |
| `AI入力補助` | 重量、サイズ、Shipping Policy、HTS候補の入力補助 |
| `出品反映候補` | 価格/送料/関税/Chatwork情報を集約する候補一覧 |

## 安全設計

- 既存シートの削除なし。
- 既存列の削除なし。
- 既存データの上書きなし。
- eBay/Sellstaへの直接反映なし。
- 自動価格改定なし。
- 自動出品停止なし。
- 取得に失敗した場合も元データは変えず、`手動確認キュー` に出す。

## 使い方

1. Google SheetsのApps Scriptに `gas/carparts_price_monitor_mvp.gs` を別ファイルとして追加する。
2. シートを再読み込みする。
3. メニュー `カーパーツ価格監視` を開く。
4. `初期セットアップ` を実行する。
5. 入力用シートを開いた状態で `現在のシートから監視マスター作成` を実行する。
6. `価格監視マスター` で10行程度を選択し、`選択行だけ価格確認` を実行する。
7. 問題なければ50件、100件へ増やす。
8. 最後に `今日の確認分を実行` で分散巡回する。

送料・関税・Chatwork側:

1. Apps Scriptに `gas/carparts_cost_control_mvp.gs` も別ファイルとして追加する。
2. `gas/carparts_price_monitor_mvp.gs` の更新版と一緒に使うと、再読み込み時に `カーパーツコスト監視` メニューも表示される。
3. `カーパーツコスト監視` から `初期セットアップ` を実行する。
4. `eBay注文取込` にeBay注文データを貼り、`eBay注文取込を整理` を実行する。
5. `部品商請求明細` に部品商共有スプレッドシートの請求明細を貼り、`部品商請求明細を再計算` を実行する。
6. `送料差額チェック` に推定送料/実送料を貼り、`送料差額チェックを再計算` を実行する。
7. `関税チェック` に届いた関税請求書の内容、申告価格、想定率、請求額を貼り、`関税チェックを再計算` を実行する。
8. `注文コスト突合を再作成` を実行し、注文ごとの粗利と利益率を見る。
9. `Chatwork情報取込` の本文列にコミュニティ投稿を貼り、`Chatwork貼付データを解析` を実行する。
10. `出品反映候補` に集まった候補を人間が確認してから、価格改定や出品停止へ進める。

## 初期設定

`設定` タブの初期値:

| 設定名 | 初期値 | 意味 |
|---|---:|---|
| `PRICE_CHECK_CYCLE_DAYS` | 14 | 14日で1周 |
| `MAX_ROWS_PER_RUN` | 80 | 1回に最大80件 |
| `MIN_SLEEP_MS` | 2000 | 1URLごとに2秒待機 |
| `PRICE_UP_ALERT_RATE` | 0.03 | 3%以上値上がりで警告 |
| `PRICE_UP_ALERT_YEN` | 100 | 100円以上値上がりで警告 |
| `MIN_MARGIN_RATE` | 0.2 | 利益率20%下限。利益計算接続後に使用 |
| `PURCHASE_FACTOR` | 0.85 | モノタロウ価格 x 85% |
| `SOURCE_MONOTARO_URL_COL` | 2 | B列 |
| `SOURCE_MONOTARO_PRICE_COL` | 3 | C列 |
| `SOURCE_PACK_QTY_COL` | 4 | D列 |
| `SOURCE_EBAY_BASE_URL_COL` | 8 | H列 |

5,000件を14日で1周する場合:

- 1日あたり約358件。
- 1回80件なら1日5回程度。
- 30日で1周に落とす場合は `PRICE_CHECK_CYCLE_DAYS` を30に変更する。

## 候補表示の意味

候補表示とは、システムが勝手に価格改定や出品停止を実行するのではなく、別タブに「対応候補」として一覧化すること。

例:

| 条件 | 表示 |
|---|---|
| モノタロウ価格が上がった | 販売価格見直し候補 |
| 価格上昇率が3%以上 | 販売価格見直し候補 |
| 取扱終了 | 出品停止候補 |
| 在庫切れ | 要確認 / 出品停止候補 |
| 価格取得失敗 | 手動確認キュー |

## 既知の制限

- モノタロウのログイン価格、キャンペーン適用後価格、部品商の実仕入価格は取得しない。
- ページ構造変更により価格抽出に失敗する可能性がある。
- 複数バリエーションや入数表記の違いは、最初は手動確認キューで扱う。
- 利益率計算は、販売価格、送料、手数料、Shipping Policyとの接続後に精度を上げる。
- サイズ/重量AI推定とShipping Policy自動候補は次フェーズ。
- Chatwork API連携は未実装。初期MVPでは手動貼付解析を使う。
- 関税率の自動取得は未実装。USITC/CBPを確認したうえで候補率を入力する。
- 部品商共有スプレッドシートの直接編集はしない。初期MVPでは貼付または参照用コピーで確認する。
- 注文コスト突合は `注文ID + SKU` を第一キーにする。関税請求書にSKUがない場合は注文ID単位で仮紐づけし、複数SKU注文は要確認にする。
