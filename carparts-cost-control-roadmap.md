# カーパーツ赤字防止・コスト監視ロードマップ

最終更新: 2026-05-27

## 結論

追加で出てきた課題は、すべて「売れたあとに赤字・欠品・請求ミスに気づく」問題につながっている。

最短で守るべき流れは、次の6つを同じ候補リストへ集約すること。

1. モノタロウ価格と取扱状態の変化。
2. 推定送料と実送料の差額。
3. 部品商の請求明細にある実仕入価格と実送料。
4. 届いた関税請求書の請求額。
5. eBay注文データにある注文国、売上、購入者支払送料、Shipping Policy。
6. Chatworkコミュニティ内の品切れ・値上げ・廃番情報。

最初から自動価格改定や自動出品停止までは行わない。まずは `出品反映候補` に集め、人間が確認してから価格改定、出品停止、Shipping Policy変更、仕入先確認へ進める。

三神さんの運用では、売れたカーパーツは部品商が直接お客様へ発送する。そのため、注文ごとの実績確認は次の3点を突合するのが最も正確。

- eBay注文データ: 注文ID、SKU、販売価格、購入者支払送料、注文国、Shipping Policy。
- 部品商請求明細: 注文ID、SKU、実仕入価格、実送料、直送先国、追跡番号。
- 関税請求書: 注文ID、請求書番号、請求関税額、通関/立替手数料、根拠URL/保存先。

## 追加課題の整理

| 課題 | いま起きる問題 | 改善後の状態 |
|---|---|---|
| 推定送料と実送料の差 | Shipping Policy選択ミスや重量推定ミスで赤字 | 差額が大きいSKUを自動で候補化 |
| 部品商請求明細の確認 | 売れた後の仕入価格/送料を注文単位で追いづらい | 注文ID/SKUでeBay注文と突合する |
| 関税請求書の確認 | 関税や通関/立替手数料が想定外でも見落とす | 請求書番号、請求額、想定額、注文IDを突合する |
| eBay注文データの確認 | 注文国、購入者支払送料、販売価格が原価確認と分かれている | 注文コスト突合で粗利と利益率を出す |
| Chatwork情報の拾い漏れ | 品切れ・値上げ・廃番情報が出品リストに反映されない | メッセージを貼付/取得して品番候補を抽出 |
| 外注リサーチの入力ミス | 税別/税込、入数、送料Policy、取扱状態でミス | 入力補助列と要確認理由を表示 |
| 既存出品の赤字化 | 古い価格・古い送料のまま販売される | 見直し候補をSKU単位で一覧化 |

## 推奨タブ構成

既存の入力用シートは壊さず、追加タブで管理する。

| タブ | 目的 | 初期運用 |
|---|---|---|
| `価格監視マスター` | モノタロウ価格、在庫、取扱終了、利益率を見る | 既存MVPで作成済み |
| `eBay注文取込` | eBay注文データを貼り、注文国/売上/送料を整理 | Seller Hub Reports等から貼付 |
| `部品商請求明細` | 部品商が直送した注文ごとの仕入価格/送料を貼る | 部品商共有スプレッドシートから貼付/参照 |
| `送料差額チェック` | 推定送料と実送料の差額を見る | CSV/手入力貼付から開始 |
| `関税チェック` | 関税請求書の請求額と想定関税額の差を見る | 請求書到着後に確認 |
| `注文コスト突合` | eBay注文、部品商請求、関税請求を突合して粗利を見る | 注文ID/SKUで自動集計 |
| `Chatwork情報取込` | 品切れ、値上げ、廃番、納期未定を抽出 | まず手動貼付、後でAPI/Webhook |
| `AI入力補助` | 重量、サイズ、Shipping Policy、HTS候補を表示 | AI候補は確定扱いにしない |
| `出品反映候補` | 価格改定、出品停止、Policy変更の候補を集約 | ここを毎日/週次で確認 |

抽出したい情報:

- 品切れ、欠品、在庫切れ。

Chatwork APIの注意:

- メッセージ取得は最大100件が前提。
- 過去の全メッセージを一気に遡る運用は期待しない。
- `force=0` の差分取得、またはWebhookで新着を拾う設計にする。
- APIトークンはシートやGitHubに保存しない。Script Propertiesなど秘密情報領域で管理する。

## AI入力補助で入れる列

| 列 | 役割 |
|---|---|
| AI_部品カテゴリ | センサー、スイッチ、フィルター、モーターなど |
| AI_推定重量kg | 送料Policy候補の材料 |
| AI_推定梱包サイズ | 小型/中型/大型、または縦横高さ |
| AI_信頼度 | 高/中/低 |
| 推奨Shipping Policy | 送料マスターから候補表示 |
| 推定送料 | 出品前の利益計算に使う |
| 送料リスク | 重い/大きい/割れ物/長尺/不明 |
| HS/HTS候補 | 関税チェックの候補 |
| 要確認理由 | 人間が見るべき理由 |
| 出品OK | OK/要確認/保留 |

AIに任せるところ:

- 商品名、型番、カテゴリからサイズ/重量候補を出す。
- 類似カテゴリの過去実績からShipping Policy候補を出す。
- Chatwork本文から品番、値上げ、品切れ、廃番の候補を抽出する。
- 関税分類の候補と確認観点を出す。

人間が最終判断するところ:

- 型番一致。
- 適合/互換。
- 関税分類。
- Shipping Policy確定。
- 価格改定。
- 出品停止。

## 短期改修ロードマップ

### Day 1

- 既存入力用シート、出品リスト、eBay注文データ、部品商請求明細、Shipping Policyマスターの列を確認する。
- 追加タブを作る。
- `出品反映候補` にすべての警告を集約するルールを決める。

### Day 2

- `eBay注文取込` と `部品商請求明細` を作る。
- 注文ID/SKUで突合できるか10件テストする。

### Day 3

- `送料差額チェック` を作る。
- eBay注文レポート、部品商請求明細、実送料データを貼れる形にする。
- 差額1,000円以上または20%以上を候補化する。

### Day 4

- `関税チェック` を作る。
- 届いた関税請求書から、請求書番号、注文ID、請求関税額、通関/立替手数料を貼れる形にする。
- 申告価格、HTS候補、想定率、請求額の差額を計算する。
- 関税が利益率を下げる商品を候補化する。

### Day 5

- `注文コスト突合` を作る。
- eBay注文、部品商請求、関税請求を注文ID/SKUで結び、実粗利と利益率を出す。
- 利益率下限割れを `出品反映候補` へ上げる。

### Day 6

- `Chatwork情報取込` を作る。
- まず手動貼付で、品番・値上げ・品切れ・廃番を抽出する。
- Chatwork API連携は、トークン管理方法を決めてから進める。

### Day 7

- `AI入力補助` を作る。
- サイズ、重量、Shipping Policy、HTS候補、要確認理由を表示する。
- 初回はAI候補だけで、確定値として使わない。
- 既存出品リストとSKU/品番で突合する。
- 価格改定候補、出品停止候補、Shipping Policy変更候補を分ける。
- Sellsta/eBay更新CSVの列案を作る。
- 外注さん向けマニュアルに反映する。
- 10件、50件、100件の順でテストする。
- 毎週の確認手順を固定する。

## 2週間巡回の考え方

現在約5,000件なら、14日で1周する場合は1日約358件。

ただし、すべてを同じ頻度で見る必要はない。

| ランク | 対象 | 推奨頻度 |
|---|---|---|
| A | 売れている、利益が薄い、仕入不安あり | 7-14日 |
| B | 通常商品 | 14-30日 |
| C | 低回転、保留、出品前候補 | 30-60日 |

設定で変えられるようにする項目:

- 価格監視の1周日数。
- 1回処理件数。
- 送料差額アラート金額。
- 送料差額アラート率。
- 関税差額アラート金額。
- 関税差額アラート率。
- 利益率下限。
- Chatworkキーワード。

## 出品数を増やして安定売上にする流れ

1. 既存出品の赤字候補を先に止める。
2. Aランク商品だけを優先して増やす。
3. 出品前に、仕入価格、送料、関税、基準セラー総額を確認する。
4. 出品CSVは10件単位でテストしてから増やす。
5. 外注さんは手入力ではなく、AI候補とCSVの例外確認に寄せる。
6. 売れた後のeBay注文データ、部品商請求明細、関税請求書、返品/キャンセル理由を次回出品判断へ戻す。

## リスクと回避策

| リスク | 重要度 | 回避策 |
|---|---|---|
| 自動取得がモノタロウ規約やページ変更に影響される | 高 | 低頻度、手動メニュー実行、失敗時は手動確認、ログイン価格は対象外 |
| Chatwork過去情報を取り切れない | 中 | 差分取得またはWebhook、新着から蓄積、初回は手動貼付 |
| 関税分類をAIが間違える | 高 | AIは候補まで、HTS/CBP確認、明細と根拠URLを残す |
| eBay注文と部品商明細が突合できない | 高 | 注文ID/SKU/追跡番号のどれで紐づくかを固定する |
| 関税請求書にSKUがない | 中 | 注文ID単位で仮紐づけし、複数SKU注文は要確認にする |
| 推定重量が外れる | 高 | 実送料差額で学習、信頼度低は要確認、大型/高額は人間確認 |
| 候補を自動反映して誤改定する | 高 | 自動価格改定・自動出品停止はしない |
| 外注さんが候補を確定値と誤解する | 中 | マニュアルに「AI候補は要確認」と明記 |
| CSVで大量誤出品する | 高 | 10件テスト、結果ファイル確認、承認後アップロード |

## 参照した公式情報

- Chatwork API: `GET /rooms/{room_id}/messages` は最大100件取得、`force=0` では差分取得。https://developer.chatwork.com/reference/get-rooms-room_id-messages
- Chatworkヘルプ: APIで全メッセージを遡って取得する運用はできず、定期差分取得またはWebhookが推奨。https://help.chatwork.com/hc/ja/articles/900002210763-API%E3%81%A7%E5%85%A8%E3%83%A1%E3%83%83%E3%82%BB%E3%83%BC%E3%82%B8%E3%81%AE%E5%8F%96%E3%82%8A%E8%BE%BC%E3%81%BF%E3%81%AF%E3%81%A7%E3%81%8D%E3%81%BE%E3%81%99%E3%81%8B
- USITC HTS: HTSは米国輸入品の関税率と統計カテゴリを確認する基準。https://hts.usitc.gov/
- CBP: 最終的な関税判断はCBP側で行われ、分類と関税率の判断は複雑。https://www.cbp.gov/trade/programs-administration/determining-duty-rates
- CBP Internet Purchases: 配送会社の見積りと実際の関税/手数料が分かれる場合がある。https://www.cbp.gov/trade/basic-import-export/internet-purchases
- eBay Seller Hub Reports: CSV/Excelで出品追加、修正、再出品、終了、レポートのダウンロード/スケジュールが可能。https://www.ebay.com/help/selling/file-exchange/seller-hub-reports?id=4096
- eBay Inventory Onboarding Guide: Business Policiesを使う場合、Payment/Return/Shipping policyを作成し、テンプレートの列へ割り当てる。https://pages.ebay.com/sh/reports/help/create-listings-bulk/
- モノタロウ利用規約: 掲載価格、仕様、取扱は事前通知なく変更/中止される場合がある。掲載情報の扱いとサイト負荷に注意する。https://www.monotaro.com/main/rule/
